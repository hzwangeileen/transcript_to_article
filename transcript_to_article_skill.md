# Skill：活动/访谈转录稿编译为文章

## 适用场景

用户上传一份飞书妙记或其他工具导出的语音转文字稿（中文或英文，说话人标注为"说话人1""说话人2"或直接标注人名），需要将其编译成结构完整的中文文章，最终输出 Word 文档（.docx）。

本 skill 适用于以下场景：
- 活动讨论、圆桌会议的记录整理
- 一对一或多人访谈的内容编译
- 演讲或分享会的文字整理

与播客编译 skill 的主要区别：输出是**以观点为主线的文章**，而非问答体对话；不区分"主持人/嘉宾"标签，重点聚焦在内容本身。

---

## 工作流程

### 第一步：输出结构预览，等待确认

收到转录稿后，**只输出以下三项内容**，然后停下来等待用户确认，不提前写任何正文。

在整理结构之前，先在内部完成转录错误的识别和修正（飞书等工具对英文专有名词识别很差，需要根据语境和公开知识主动修正，比如把"cloud"修正为"Claude"，把"Leyden Space"修正为"Latent Space"）。修正项在输出章节结构时一并列出。

**1. 转录错误修正清单**

以表格形式列出所有识别到的转录错误，格式为：原文错误词 → 修正后。覆盖人名、公司名、产品名、专业术语等所有专有名词，不遗漏。

**2. 章节结构**

章节**不按对话顺序划分**，而是围绕核心观点重新组织，归纳为 5 到 8 章。章节之间要有前后逻辑，整体读下来是一篇结构完整、叙述清晰的文章，而不是把话题随机拼在一起。

**分类和排序逻辑**：
- 相同或相近的话题必须放在同一章节，不要分散在不同章节重复出现
- 章节之间的顺序要服务于文章想要传达的核心信息。例如，介绍一家公司时，可以先写行业背景和痛点，再写公司的产品和技术，然后是商业模式，最后是竞争格局和未来判断。这只是参考，具体顺序应根据内容本身决定，不必套用固定模板
- 如果是行业趋势类讨论，可以先建立共识性背景，再展开各方的分歧和判断
- 文章的开头章节应当能让不了解背景的读者快速进入语境；结尾章节应当有一定的收束感，而不是突然中断

每个章节标题应当概括该章节的核心论断，而非只是贴标签。参考风格：
- "开源模型下限达标，才是 Token 爆炸的真正引线"
- "高价值 Agent 必须由行业专家主导"
- "Excel 是生产力的放大和延伸，而不是被替代的对象"
- "Google TPU 会'革英伟达的命'吗？"（疑问句也可，适合存在明显分歧的话题）

以列表形式输出章节结构，每条附一句话说明该章节的主要内容。

**3. 文风示范与风格确认问题**

不要用"本文将采用……风格"这类元描述来说明文风，而是**直接用目标文风写一段正文**，选取转录稿中一个具体的论点或场景，按照拟采用的风格写出来。示范段落长度约 100 到 150 字，足以让用户感受到节奏、语气和论断方式，从而判断是否符合预期。

同时，**主动询问用户以下问题**，以便在生成前对齐预期：

- 文章是否需要带有"我们认为/我们看好"这类作者判断视角，还是保持客观陈述？

如果用户有参考风格样本（如上传了参考文章 PDF），应提取其文风特征作为示范段落的写作依据，无需再单独询问。

以上三项输出完之后，**停下来等用户反馈**。未收到确认前不进行下一步。

---

### 第二步：收到确认后，直接生成完整 Word 文档

用户确认结构和文风后，按照确认后的章节结构写完全部章节，直接生成完整 .docx 文件，不再逐章停顿确认。

Word 文档结构依次为：
1. 标题选项（3 到 5 个备选，风格各异）
2. 摘要金句选项（2 到 3 句，适合放在标题下方作导语）
3. 引言（根据全文内容写成，放在带紫色边框的方框中，约 200 到 300 字，末尾附核心观点列表 3 到 5 条）
4. 正文（按章节编号展开）

## 格式规范

**章节标题**

编号格式为 `01`、`02` 以此类推，紫色编号加黑色标题文字，字号大于正文。**章节编号和大标题之间不加任何副标题行**，直接进入子标题和正文。

**章节内子标题**

每个章节内部，按照不同的论点或子话题分类，设置若干蓝色/紫色子标题（颜色用 `#5B4FCF`），不编号。子标题应当是一个独立的论断或问题，而非泛泛的分类词。

示例：
- "Google 在 pre-training 算力上首次追平 OpenAI"（论断型）
- "Google TPU 会'革英伟达的命'吗？"（疑问型，适合存在分歧的子话题）
- "Cowork 和 Claude Code in Excel 会打开'10x 微软'市场"（结论型）

**正文段落**

纯文章体，不使用对话形式，不出现"某人提问""某人回答""主持人问道"等字样。一个人说了很多内容时，按逻辑拆成多个段落，每段约 3 到 5 句话，不要堆成大段。

金句加粗，其余正文不加粗。

**多方观点的呈现方式**

当一个话题存在明显的分歧或多种立场时，应当明确标出立场标签，方式参考：
- "看多的一方认为……看空的一方认为……"
- "激进派：……保守派：……"
- "有观点认为……但也有不同的声音认为……"

不同立场的观点应当都完整呈现，不能只写其中一方。

**备注框**

统一使用浅紫色边框注释框，放在相关内容后面。专有名词解释、产品背景、最新动态都用这一种样式，最新动态以"最新动态："开头。只在读者真正需要背景知识的地方加，不要为了加备注而加。

备注框内容应当简洁，一般 1 到 3 句话，说清楚是什么、有什么关键背景信息即可。

## 编译规范

**内容处理**

所有内容必须完整保留，按逻辑重新组织，不做任何归纳、压缩或省略。只删掉纯口语语气词（如"嗯""啊""哈哈""like""you know"），保留所有实质内容，包括举例、旁白、插话、题外话等。**信息零损失是最高优先级，任何过度缩减都是错误。**

**篇幅基准：输出字数应与原文字数大体相当。** 原文 10,000 字，成文应在 10,000 字上下；原文 6,000 字，成文应在 6,000 字上下。中文成文天然比英文原文略短，这属正常，但不得大幅压缩。如果成文比原文短很多，说明内容被遗漏了，必须补全。

可以对原文的表达做编辑性调整，让意思更清晰、读起来更流畅，但调整只能发生在"怎么说"的层面，不能影响"说了什么"。每一个信息点、每一个例子、每一段推理都必须完整呈现。不得用一句话概括原文的一大段论述，不得把多个例子合并成"他举了几个例子"。

飞书转录错误主动修正，根据语境和公开知识判断，不照搬原文错误词。

**数字与非公开信息的来源标注**

涉及数字或具体数据时，需区分两种情况：

- **行情类、人尽皆知的公开数据**（如市场通行的 GPU 价格区间、某产品的公开融资金额、已被广泛报道的市场规模等），可以直接陈述，无需标注来源。
- **非公开数据，或来自特定人士的内部观察、一手经验、估算判断**（如某人透露的团队人均 Token 消耗、某公司内部的测算数字、某位嘉宾给出的具体比例或预测值等），必须在行文中保留信息来源，例如："据某某透露""有观点认为""某人分享的数据显示"。

判断原则：**这个数字是否是说话人的个人观察或独家信息？** 如果是，就必须标注出处；如果这个数字在公开渠道能轻松查到，则不需要。

**内容组织原则**

- 不按对话顺序写，而是围绕核心观点逻辑展开
- 不出现"某人提问""某人回答"等对话痕迹
- 提问者的重要背景信息和引导性内容可以保留，但融入正文叙述而非以问题形式呈现
- 一个信息点只在一个章节出现，不重复
- 不同说话人就同一话题发表的观点可以合并在同一段落叙述；但各自的独特数据、独家观察、个人判断需保留归属，不能混为一谈
- 同一章节内部，如果同时涉及多个子话题（比如多个产品、多家公司、多种观点），每个子话题单独开一个子标题，不要混在一起
- 转录错误在成文中直接使用修正后的版本，无需标注，修正清单在第一步结构预览时一并输出

**语言风格**

默认文风：研究报告体。结论前置、判断明确，关键论断加粗落地，数据和结论紧密绑定。行文中保留说话人归属（如"Cameron 指出""Jason 描述"），但不加入"我们认为/我们看好"等作者判断语句，保持客观陈述。

如果用户上传了参考风格样本，应提取其文风特征，在文风说明中具体说明将如何对齐，并以此为准覆盖默认风格。

具体要求：
- **每个句子都要有完整的主谓宾结构**，不能省略主语或动词，避免出现"通过……实现了……"这类没有明确施动者的句式
- 遇到专业概念或行业背景，在行文中自然带出解释，或在备注框里补充，不要假设读者已经懂
- 句子简洁有力，不堆砌形容词，不绕弯子
- 每段约 3 到 5 句，节奏感强，不拖沓
- 论断要明确落地，不要用模糊的"可能""也许"弱化所有观点，但确实是推测性判断时保留"有观点认为"等归因表述
- 用词准确，专业术语保留英文原文（Token、Agent、Infra、Scaling Law、pre-training、post training 等），不强行翻译，但第一次出现时可以在括号里简单说明是什么
- 不用书名号或特殊引号，统一用普通双引号 ""
- 不用破折号，改用逗号或句号
- 不用「」，改用普通双引号 ""
- 不写"值得注意的是""不可否认""不言而喻""不得不说""毋庸置疑"这类 AI 味或套话式措辞
- 不写"这意味着……""这表明……""这说明……"连续出现超过两次，避免行文机械重复
- 数据和案例紧跟论点出现，不要单独另起一段做孤立陈述

## 校对要求

在生成完整文章之前，需要做以下两类校对。

**对转录稿的校对**

检查成文是否有对转录内容理解有误的地方，包括：是否有过度夸张或过度总结的表达，是否有信息遗漏。

**对公开信息的校对**

搜索公开网络信息，逐一核查以下内容是否与事实相符：
- 专有名词（人名、公司名、产品名、论文名及作者）
- 数字（规模、成立时间、融资金额等）
- 事件描述（发布时间、功能描述等）

发现错误时，在文中直接修正，并在结构预览阶段的纠错列表中一并标注说明。

## 技术实现

使用 docx npm 包（JavaScript）生成文件，输出到 `/mnt/user-data/outputs/`，用 `present_files` 工具提供下载链接。生成后用 `validate.py` 校验，确认段落数大于零且无报错。

颜色参数：章节编号颜色 `#5B4FCF`，子标题颜色 `#5B4FCF`，备注框背景 `#F5F3FF` 左边框 `#9B8FE0`，引言框背景 `#EDE9FF` 全边框 `#5B4FCF`。
